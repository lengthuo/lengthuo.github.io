提交代码规范

当公司没有具体的规定的时候，就必须严格遵守自己的规定

实例

feat(module):(JRY-120) 后台订单列表

定义

<type>(<scope>):<jira:id><subject>
    <BLANK LINE>
<body>
   <BLANK LINE>
<footer> 

jira:id : 
       JIRA 号(没有填:JRY-00)
type：
       下面有几个我们常用到的
      . feat :新功能 
      . fix :修复bug  
      . doc : 文档改变
      . style : 代码格式改变
      . refactor :某个已有功能重构
      . perf :性能优化
      . test :增加测试
      . build :改变了build工具 如 grunt换成了 npm
      . revert: 撤销上一次的 commit 
      . chore 维护
scope：
       用来说明此次修改的影响范围 可以随便填写任何东西，推荐使用
       all ：表示影响面大 ，如修改了网络框架  会对真个程序产生影响
       loation： 表示影响小，某个小小的功能
       module：表示会影响某个模块 如登录模块、首页模块 、用户管理模块等等  
       
subject:
       用来简要描述本次改动，概述就好了
body:
       具体的修改信息 应该尽量详细
footer：
       放置写备注啥的。

git命令大全

2017/11/3
posted in git 

git 常用命令

git reset 将当前HEAD复位到指定状态


git reset HEAD~1                    保留修改，不保留commit状态
 
git reset --soft HEAD~1             保留修改，保留commit的状态

git reset --hard HEAD~1             不保留修改，直接跳转到上个commit的状态

设置提交人信息

// 设置全局
git config --global user.name "Author Name"
git config --global user.email "Author Email"

// 或者设置本地项目库配置
git config user.name "Author Name"
git config user.email "Author Email"

修改最后一次提交信息


Git commit –amend  "xxxx"

修改文件权限禁止git捕获


git config core.filemode false

仅从git仓库中删除指定文件，而保留其在working tree中

git rm --cached <文件名> # 仅从git仓库中删除<文件名>而在working tree中保留

git历史 展示

git log --oneline --graph --decorate --all

多次commit合并成


git rebase -i  id  （合并这个id之前的）

其中，-i 的参数是不需要合并的 commit 的 hash 值.


当前我们只要知道 pick 和 squash 这两个命令即可。

1. pick 的意思是要会执行这个 commit
2. squash 的意思是这个 commit 会被合并到前一个commit

 保存退出之后就会弹出commit message 的编辑界面
 编辑commit的message即可

git log --oneline --graph --decorate --all
